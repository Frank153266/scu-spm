# AI技术应用验证报告

**验证场景**：AI驱动API测试用例与数据生成

#### 1. 验证场景与目标

- **选定场景**：针对系统设计文档(SDD)中定义的“代码提交”接口（`POST /api/submission`），验证AI工具在生成该接口的自动化测试用例时的效率和质量。
- **验证目标**：
  1. **效率**：对比人工编写与AI辅助编写测试用例所花费的时间。
  2. **质量**：对比两者在功能点覆盖、异常场景覆盖、边界值覆盖上的差异。

#### 2. 工具选型与执行过程

- **AI工具**：**Postman + Postbot AI** (基于OpenAPI规范生成测试集合)。
- **输入**：
  1. **OpenAPI规范**：从SDD 4.4节手动提炼的`/api/submission`接口规范（包含请求体结构、响应码）。
  2. **自然语言Prompt**：“Generate a comprehensive test suite for this API endpoint. Include tests for success (200), authentication failure (401), invalid input (400), and server error (500). Use dynamic variables for user IDs and problem IDs.”
- **执行过程**：
  1. 在Postman中创建一个新的API集合。
  2. 手动定义`/api/submission`接口（模拟从OpenAPI导入）。
  3. 选中该接口，点击“Generate tests with Postbot”。
  4. 在弹出的对话框中输入上述Prompt。
  5. Postbot AI在10秒内生成了包含6个请求的文件夹，每个请求都附带了前置脚本（如设置JWT）和测试断言（如`pm.expect(pm.response.code).to.eql(200)`）。
  6. 人工对生成的脚本进行审查并微调（如修正一个不准确的断言消息）。
- **人工方式（对照组）**：一名熟悉Postman的测试人员，根据需求文档F-016/017，从零开始编写相同功能的测试集合，耗时约45分钟。

#### 3. 结果对比与分析

| 对比维度            | 人工方式 (对照组)                                 | AI辅助方式 (实验组)                                          | 差异分析                                                     |
| :------------------ | :------------------------------------------------ | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **耗时**            | **45分钟** (包括查文档、设计用例、编写脚本、调试) | **12分钟** (1分钟定义Prompt, 10秒生成, 10分钟审查微调)       | **效率提升73%**。AI节省了设计用例结构和编写样板断言的时间。  |
| **功能点覆盖**      | 4个(正常提交、未登录、缺少参数、题目不存在)       | 6个(正常提交、未登录、缺少userId、缺少代码、提交空代码、数据库连接失败Mock) | AI生成的异常场景更丰富，不仅覆盖了业务错误(题目不存在)，还覆盖了参数级错误和底层系统错误(Mock)。 |
| **边界/异常值覆盖** | 覆盖了“代码长度=10001行”(手动构造)                | 除代码长度外，还生成了“代码含危险命令(`rm -rf /`)”、“提交间隔<1秒”等安全与并发场景。 | AI基于其对软件测试最佳实践的训练，生成了更多安全性和限流相关的测试点，超出了当前需求文档的直接描述。 |
| **脚本可用性**      | 100%，符合预期。                                  | 95%，生成的断言逻辑准确，但有一个请求的URL路径变量未正确替换，需要人工修正。 | AI未能完全理解动态路径参数，但错误易于识别和修复。总体可用性非常高。 |

#### 4. 改进建议

1. **过程集成**：将“AI生成API测试”作为**强制节点**集成到CI/CD流水线中。每当后端API代码变更时，应重新触发AI生成测试用例并与已有用例合并，确保测试集始终与最新代码同步。
2. **Prompt工程化**：建立团队的“Prompt模板库”。例如，针对核心CRUD接口，固化为一个通用的测试生成模板，只需填入接口名称和关键字段，即可自动生成80%的标准化测试用例。
3. **工具链深化**：尝试更智能的工具如 **Keploy** 或 **EvoMaster**，它们可以通过流量录制或搜索算法，自动生成更高代码覆盖率（如分支覆盖、突变覆盖）的测试用例，而不仅限于API层面的E2E测试。

#### 5. 验证结论

本次原型验证结果清晰地表明，在API测试用例生成环节引入AI工具是**成功且高效的**。它不仅将测试脚本编写时间**缩短了70%以上**，更重要的是，它生成了比人工更全面、更具探索性的测试场景（特别是安全和边界条件），这对于保障“智能在线编程练习平台”的核心——判分系统的稳定性和安全性，具有极高的价值。建议在项目中全面推广该改进方案。
